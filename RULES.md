## 一、核心流程（每轮循环）

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  ① 读上轮交接文档 (docs/HANDOVER/round_N.md)         │
│  ② 读当前轮工作日志 (ROUND_LOG.md)                   │
│  ③ 审阅本轮计划项目 (PLAN.md → 当前 Phase → Round)  │
│  ④ 审阅架构文档 (MMI统一架构设计.md)                 │
│     → 对齐检查：方向有没有偏离？                    │
│     → 偏离则纠正，不偏离继续                         │
│  ⑤ 确认后开始工作                                    │
│       ↓                                              │
│  ⑥ 每完成一个子任务 → 更新 ROUND_LOG.md             │
│       ↓                                              │
│  ⑦ 全部子任务完成 → 跑测试                          │
│       ↓                                              │
│  ⑧ 写交接文档 (docs/HANDOVER/round_N.md)            │
│       ↓                                              │
│  ⑨ git commit → 进入下一轮循环                      │
│                                                      │
└──────────────────────────────────────────────────────┘
```

## 五、编码规范

### 5.1 基本原则
- **不破坏现有接口**：所有 CLI 命令向后兼容
- **渐进式改造**：新功能用新模块，不直接改旧代码（除非必要）
- **测试先行**：先写测试，再写实现（至少同步）
- **类型标注**：所有公共函数 + 参数 + 返回值

### 5.2 命名约定
- 记忆引擎：`mmi/core/xxx.py`
- Agent调度：`mmi/agent/xxx.py`
- 类：`PascalCase`
- 函数：`snake_case`
- 常量：`UPPER_SNAKE`
- 私有：`_leading_underscore`

### 5.3 提交规范
```
<type>: <简短描述>

<详细说明（可选）>

Ref: Round X.Y
```

type 取值：`feat` `fix` `refactor` `test` `docs` `chore`

### 5.4 分支策略
- `main` — 主线，不动
- `feature/xxx` — 当前工作分支
- 每轮结束后 commit，不 force push

## 六、质量门禁

每轮结束前必须通过：

| 门禁 | 命令 | 标准 |
|---|---|---|
| 代码风格 | `ruff check mmi/` | 0 error |
| 单元测试 | `pytest tests/ -x` | 全部通过 |
| 文档检查 | PLAN.md + ROUND_LOG.md 已更新 | — |

---

## 七、风险控制

1. **每次只改一个模块**，不跨模块大改
2. **改动前备份思路**：在 ROUND_LOG.md 记录"准备怎么改"
3. **出错回退**：`git stash` → 分析原因 → 重新来
4. **不确定就问**：不要猜，主动确认

---

## 八、禁止事项

- ❌ 禁止在 main 分支直接修改
- ❌ 禁止跳过测试直接提交
- ❌ 禁止删除旧代码而不做兼容（标记 deprecated 即可）
- ❌ 禁止一轮改超过 3 个模块
- ❌ 禁止无交接文档就跳到下一轮
